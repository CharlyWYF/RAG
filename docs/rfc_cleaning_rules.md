# RFC 清洗与切分规则

## 1. 目标

RFC 文档通常篇幅较长，包含目录、页眉页脚、分页信息、附录、参考文献等与问答任务关联较弱的内容。如果直接将原始 RFC 文本整体切分并建库，容易带来以下问题：

- 检索结果混入大量无关内容
- 章节边界丢失，引用难以定位
- chunk 语义不完整，影响回答质量
- 检索召回相关但不直接回答问题的文本片段

因此，本项目对 RFC 文档采用“轻量清洗 + 结构优先切分”的处理策略，在尽量保留标准原文语义的前提下，提升其对 RAG 问答任务的适配性。

## 2. 总体处理原则

RFC 处理遵循以下原则：

- 不改写原文核心语义
- 优先保留章节结构信息
- 优先删除与问答弱相关的噪音内容
- 优先保证 chunk 的语义完整性与可引用性
- 保证每个 chunk 可追溯到具体 RFC 和章节

## 3. 输入与输出

### 3.1 输入

输入为原始 RFC 文本文件，例如：

- `data/protocols/tcp/rfc793.txt`
- `data/protocols/dns/rfc1035.txt`
- `data/protocols/http/rfc2616.txt`

### 3.2 输出

清洗后的文本输出到：

- `data/protocols/cleaned/`

建议按协议和 RFC 编号组织输出，例如：

- `data/protocols/cleaned/tcp/rfc793.cleaned.txt`
- `data/protocols/cleaned/dns/rfc1035.cleaned.txt`
- `data/protocols/cleaned/http/rfc2616.cleaned.txt`

## 4. 清洗规则

### 4.1 需要删除的内容

以下内容原则上删除或排除出建库范围：

- 页眉
- 页脚
- 分页页码
- RFC 首页中与问答无关的发布说明
- 目录（Table of Contents）
- 参考文献（References）
- 致谢（Acknowledgements）
- 作者信息（Author's Address / Authors' Addresses）
- 附录（Appendix），若与核心协议问答关系不大可整体排除
- IANA Considerations（第一版可排除）
- Security Considerations（第一版可视情况排除）
- 空白行过多导致的格式噪音
- 由于换页产生的断裂行和重复行

### 4.2 需要保留的内容

以下内容应尽量保留：

- 章节编号
- 章节标题
- 协议定义
- 字段说明
- 报文格式
- 状态转换描述
- 连接建立与终止流程
- 查询与响应机制
- 约束条件
- 错误处理机制
- 与协议行为直接相关的规范性描述

### 4.3 谨慎处理的内容

以下内容不要简单粗暴删除，应根据协议类型判断其价值：

- `Security Considerations`
- `IANA Considerations`
- 示例报文
- 图表与 ASCII 格式结构图
- 状态机描述
- 字段位布局说明

如果这些内容对回答“字段含义”“流程机制”“版本差异”“安全性约束”等问题有帮助，应保留。

## 5. 结构切分策略

## 5.1 一级切分：按章节结构切分

优先识别 RFC 中的章节标题，例如：

- `1. Introduction`
- `2. Overview`
- `3.1 Header Format`
- `4.2.2 Connection Establishment`

按章节层级切分，而不是一开始就按固定长度切分。

每个章节单元至少包含：

- 章节编号
- 章节标题
- 对应正文内容

### 5.2 二级切分：对超长章节再次切分

若某章节正文过长，则在保留章节上下文的基础上继续切分，推荐方式如下：

- 优先按自然段切分
- 若自然段累计过长，再按固定 token/字符长度切分
- 保留一定 overlap，以避免上下文断裂

建议参数可初步设为：

- `chunk_size`: 600 \~ 1000 tokens
- `chunk_overlap`: 80 \~ 150 tokens

具体参数可在实验中比较。

### 5.3 不建议的做法

以下做法不建议直接采用：

- 对整篇 RFC 直接按固定长度切块
- 切分时丢弃章节标题
- 将多个不相关小节合并成一个大 chunk
- 在清洗阶段大量改写原文

## 6. Chunk 元数据设计

每个 chunk 建议至少包含以下元数据字段：

- `doc_id`：文档唯一标识，例如 `tcp_rfc793`
- `protocol`：协议名，例如 `tcp`
- `rfc_number`：RFC 编号，例如 `793`
- `source_file`：原始文件路径
- `cleaned_file`：清洗后文件路径
- `section_id`：章节编号，例如 `3.4`
- `section_title`：章节标题
- `chunk_id`：chunk 唯一编号
- `source_type`：固定为 `rfc`
- `source_url`：RFC 官方链接
- `text`：chunk 正文内容

如果后续要增强可观测性，还可以增加：

- `char_count`
- `token_estimate`
- `parent_section`
- `chunk_index_in_section`

## 7. 建议的处理流程

建议将 RFC 处理分成以下步骤：

1. 读取原始 RFC 文本
2. 删除页眉页脚、页码、目录等明显噪音
3. 定位正文起始位置
4. 截断参考文献、作者信息、附录等尾部弱相关内容
5. 识别章节标题并建立章节结构
6. 按章节输出清洗结果
7. 对超长章节进行二次切分
8. 为每个 chunk 生成唯一 ID 和元数据
9. 写入向量库

## 8. 第一阶段适用范围

第一阶段优先对以下 RFC 应用该规则：

- `rfc793.txt`
- `rfc1035.txt`
- `rfc2616.txt`

原因如下：

- 这三篇文档能支撑系统核心演示
- 问题类型丰富
- 适合验证清洗与结构切分策略是否有效

待第一阶段效果稳定后，再扩展到：

- `rfc791.txt`
- `rfc768.txt`
- `rfc5246.txt`
- `rfc8446.txt`
- `rfc7540.txt`

## 9. 评估关注点

RFC 清洗与切分效果可从以下角度评估：

- chunk 是否保留完整语义
- 检索结果是否能命中正确章节
- 引用是否能定位到 RFC 编号与章节标题
- 是否显著减少无关噪音片段
- 回答是否更聚焦、幻觉是否减少

## 10. 实施注意事项

- 清洗后的文本不应覆盖原始 RFC 文件
- 原始文本与清洗文本应同时保留，便于回溯与调试
- 清洗规则应尽量通用，但允许针对特定 RFC 做少量特化
- 第一版先追求稳定和可复现，不必一开始追求最优参数
- 后续实验中若发现典型错误案例，应优先从清洗规则、章节切分与 chunk 参数进行迭代
