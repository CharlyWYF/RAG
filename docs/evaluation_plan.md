# RAG 系统评估方案（论文实验版）

## 1. 评估目标

本项目旨在构建一个面向网络协议标准文档的 RAG（Retrieval-Augmented Generation）问答系统。为了系统性评估其实际效果，需要从检索有效性、回答质量、保守回答能力以及系统性能四个层面对系统进行综合分析。

考虑到本项目知识源以 RFC 标准文档为主，问题集为自定义协议问答场景，且实验语言环境为中文，因此在评估方法上不直接照搬通用开放问答 benchmark，而是采用“参考主流 RAG 评估思想 + 结合任务特征进行本地化设计”的策略。

---

## 2. 方法依据：参考 RAGAS 的评估思想

在 RAG 系统评估研究中，RAGAS 提供了较有代表性的评估思路，其关注的核心维度通常包括：

- 回答相关性（Answer Relevancy）
- 回答忠实性（Faithfulness）
- 上下文精度（Context Precision）
- 上下文召回（Context Recall）

本项目在设计评估方案时，参考了上述评估维度，但未完全依赖 RAGAS 原始框架，而是结合本项目实际情况进行了适配，原因包括：

1. 本项目测试问题集为自定义协议问答集合，而非通用 benchmark 数据集。
2. 项目知识源为 RFC 标准文档和少量私有演示协议文档，知识分布与开放域问答不同。
3. 论文实验不仅关注回答质量，还关注检索命中、保守回答能力和时延表现。
4. 在当前阶段，人工评分结合自动统计指标更适合本项目规模与毕业设计实验要求。

因此，本项目采用“自动统计指标 + 人工评分指标”的混合评估方案。

---

## 3. 评估数据集

### 3.1 测试问题集来源

本项目测试问题集由项目内部设计，覆盖以下三类：

1. **RFC 标准问题**
   - 主要围绕 DNS、HTTP、TLS、TCP、UDP、IP、ICMP、ARP 等协议文档构造
2. **保守回答测试问题**
   - 用于测试系统在证据不足场景下是否能够拒答或保守回答
3. **私有协议演示问题**
   - 以项目自定义协议 SUT 为例，测试系统对私有知识源的适应能力

### 3.2 数据组织方式

测试问题集包含结构化字段，例如：

- 题号（ID）
- 问题内容（Question）
- 协议类别（Protocol Group）
- 题型（Question Type）
- 目标文档（Target Document）
- 难度（Difficulty）
- 预期要点（Expected Keypoints）
- 是否应保守回答（Should Refuse）

这使得测试集既可用于人工评测，也可用于自动化实验统计与图表生成。

---

## 4. 评估维度与指标

### 4.1 检索有效性

该维度用于评估系统能否为问题检索到正确或高度相关的协议文档。

#### 指标 1：目标文档命中率（Target Document Hit Rate）

定义：在每道题的检索来源中，是否包含预期目标文档。

用途：
- 衡量系统是否能够将问题映射到正确协议文本
- 适合本项目的 RFC 目标文档场景

#### 指标 2：平均唯一来源数（Average Unique Source Count）

定义：每道题最终回答所涉及的去重来源文档数量的平均值。

用途：
- 反映检索覆盖面
- 辅助分析系统是否依赖单一来源，或能够进行跨文档整合

#### 可选人工指标：来源相关性评分

建议评分：
- 0 = 来源明显无关
- 1 = 来源部分相关
- 2 = 来源高度相关

用途：
- 弥补“命中目标文档”之外的细粒度评估

---

### 4.2 回答质量

该维度用于评估最终生成回答的内容质量。

由于本项目问题集为自定义协议问题，且部分问题需依赖领域判断，因此采用人工评分更合适。

#### 指标 3：正确性（Correctness）

建议评分：
- 0 = 回答错误
- 1 = 部分正确
- 2 = 回答正确

用途：
- 衡量系统回答是否符合 RFC 文本事实

#### 指标 4：完整性（Completeness）

建议评分：
- 0 = 明显不完整
- 1 = 基本完整
- 2 = 完整覆盖关键要点

用途：
- 衡量回答是否覆盖问题的主要机制、字段或对比维度

#### 指标 5：忠实性（Faithfulness）

建议评分：
- 0 = 出现明显幻觉或偏离上下文
- 1 = 基本基于上下文，但有轻微推断
- 2 = 严格依据检索上下文回答

用途：
- 对应 RAGAS 中的 Faithfulness 思想
- 评估回答是否真正“基于检索结果”而非凭空生成

---

### 4.3 保守回答能力

该维度用于评估系统在证据不足场景下的控制能力。

#### 指标 6：保守回答成功率（Refusal Success Rate）

定义：在“应保守回答”的问题中，系统实际成功触发保守回答的比例。

用途：
- 评估系统是否具备合理的拒答能力
- 避免在 RFC 证据不足时产生幻觉

#### 指标 7：误拒答情况（False Refusal）

定义：在“本不应拒答”的问题中，系统却错误触发保守回答的情况。

用途：
- 评估系统是否过于保守

#### 图形化表达：保守回答混淆矩阵

可构造 2×2 评价矩阵：

- 应保守回答 / 不应保守回答
- 系统保守回答 / 系统正常回答

该矩阵有助于直观展示系统在拒答控制上的优劣。

---

### 4.4 系统性能

该维度用于评估问答系统的交互时延和运行效率。

#### 指标 8：平均总耗时（Average Total Time）

定义：从问题进入系统到生成完整回答的平均时间。

#### 指标 9：平均首字响应时间（Average First Token Time）

定义：从问题进入系统到首次输出回答内容的平均时间。

用途：
- 反映用户主观感知的交互延迟

#### 指标 10：阶段耗时（Stage Timing）

本项目将问答流程拆分为若干主要阶段，例如：

- 加载配置
- 查询改写
- 初始化检索器
- 向量检索
- 初始化大模型
- 生成最终回答

用途：
- 分析系统耗时主要集中在哪些阶段
- 支撑后续性能优化讨论

---

## 5. 实验流程

本项目建议采用如下实验流程：

1. 运行自动化评测脚本，批量执行结构化问题集。
2. 收集自动统计结果，包括：
   - `results.jsonl`
   - `results.csv`
   - `summary.json`
3. 由研究者对回答质量进行人工评分。
4. 结合自动统计结果与人工评分结果，生成最终实验表与图。
5. 对比分析：
   - 不同协议类别上的表现差异
   - 不同问题类型上的表现差异
   - 检索能力、回答质量与时延之间的关系

## 7. 人工评分模板建议

在自动评测结果基础上，建议增加以下人工评分列：

- `correctness_score`
- `completeness_score`
- `faithfulness_score`
- `retrieval_relevance_score`
- `manual_notes`

这将形成完整的“自动统计 + 人工评分”实验闭环。

---

## 8. 方案总结

综上，本项目评估方案具有以下特点：

1. 参考了 RAGAS 的主流评估思想；
2. 根据 RFC 协议知识问答场景进行了任务适配；
3. 同时关注检索、回答、拒答与性能四个维度；
4. 兼顾自动化统计与人工评分，适合毕业论文实验分析与图表展示。
