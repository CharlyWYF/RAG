RAG Source Map / SRC Architecture

用一页看懂 /src
每个文件到底在做什么

这个页面不是简单列目录，而是把当前 RAG 项目里 src/ 的模块职责、相互关系和在整体流程中的位置直观展开。 你可以把它理解成一张“代码职责地图”：哪些文件负责配置，哪些文件负责建库，哪些文件负责检索与问答，哪些只是辅助工具。

当前 /src 结构

src/
├─ config.py
├─ ingest.py
├─ retriever.py
├─ qa.py
├─ file_ops.py
└─ presentation.py

阅读方法

核心配置层：决定模型、路径、chunk、top-k 等全局行为。
建库离线层：处理 cleaned 文档、切块、向量化并写入 Chroma。
问答在线层：负责检索器、提示词、模型调用与问答结果生成。
辅助工具层：给 app.py 提供文件处理、展示格式化等可复用逻辑。

模块职责卡片

下面每张卡片都对应一个 src 文件：先告诉你它的角色，再说明它在当前项目里具体负责什么、和哪些模块耦合。

src/config.py 配置核心

配置中心

这是整个项目的“单一配置入口”。它负责从 .env 读取关键参数，并组装成统一的 Settings 对象。

  • 读取 API Key、模型名、top-k、chunk 参数
  • 校验数值合法性，例如 overlap 不能大于 chunk size
  • 给检索、问答、建库三个流程提供统一配置来源
src/ingest.py 离线建库

索引构建引擎

这是 raw/cleaned 之后真正把文档写进向量库的地方。它处理加载文档、切块、增量比对和写入 Chroma。

  • 支持 rebuild / sync / append 三种模式
  • 支持 fixed / section / hybrid 三种 chunk 策略
  • 记录建库统计信息与阶段耗时，供前端展示与性能分析
src/retriever.py 检索入口

向量检索器构造器

这个文件只做一件事：把持久化的 Chroma 向量库包装成可直接调用的 retriever，供问答流程使用。

  • 读取当前 embedding 配置
  • 连接本地 Chroma 库
  • 按照 top_k 返回检索器对象
src/qa.py 问答核心

问答基础能力

这里承载了问答层最核心的通用能力：提示词模板、LLM 构造、embedding 构造、健康检查，以及一版通用问答流程。

  • 定义 RAG 回答提示词
  • 统一创建 ChatOpenAI 与 OpenAIEmbeddings
  • 提供 API 连通性检查与基础问答执行函数
src/file_ops.py 文件工具

文件与路径工具箱

这是从 app.py 中抽出来的纯工具模块，负责 raw/cleaned 文件处理和 .env 文件读写。

  • 读取 / 写回 .env
  • 列出 raw 文档
  • 判断某个 raw 文件是否已有 cleaned 结果
  • 解析来源路径，服务预览页与知识库管理页
src/presentation.py 展示辅助

展示格式化层

这个文件不碰业务逻辑，只负责把运行时数据整理成前端更好展示的形式，比如耗时表、阶段中文标签、预览 URL。

  • 问答阶段英文名 → 中文名
  • 建库阶段英文名 → 中文名
  • 把 timing / stats 整理成表格结构
  • 生成来源预览页链接

它们在整体流程里怎么配合

01

准备与配置

config.py 读取环境变量；file_ops.py 处理 raw/cleaned 文件与配置文件读写，为页面交互提供底层支持。

02

离线建库

ingest.py 读取 cleaned 文档，切块、向量化并写入 Chroma；同时输出文档数、chunk 数与阶段耗时。

03

在线问答

retriever.py 连接向量库；qa.py 负责提示词、模型调用和基础问答能力；presentation.py 把结果整理给页面展示。

为什么还保留 app.py 很大？

因为它仍然是 Streamlit 入口文件，负责 tab 布局、按钮交互、状态占位和运行时反馈。现在已经把纯工具逻辑开始抽离，但页面层本身还在这里。

下一步最值得继续拆什么？

最值得继续抽的是问答编排：查询改写、多路检索、去重、流式回答与 timing 汇总。这些目前还主要在 app.py 的问答 tab 中。

这个页面最想让你记住什么？

src 不是按“页面”拆的，而是按“职责”拆的。 配置、建库、检索、问答、文件工具、展示辅助各自承担清晰角色，这样后续扩展才不会让 app.py 一直膨胀。
