# RAG 项目阶段性总结与后续计划

## 1. 当前阶段已完成的工作

### 1.1 RAG 系统核心链路已经跑通

当前项目已经具备较完整的端到端 RAG 主流程，不再只是概念验证，而是一个可实际运行、可构建知识库、可执行问答和可开展实验评估的系统。

已完成的核心链路包括：

- 原始文档收集与整理
- 文档清洗
- `raw -> cleaned` 双目录规范化
- 向量库构建
- 检索式问答
- 查询改写增强检索
- 多路检索与去重
- 流式回答输出
- 首字响应时间与总耗时统计

这说明系统已经具备论文所需的基本实现基础：不仅“能设计出来”，而且“能实际运行、能构建、能回答、能评估”。

---

### 1.2 语料体系已经明确并完成扩展

#### 知识源定位已明确

项目已经明确采用：

- **RFC 标准文档** 作为正式实验的主体知识源
- 不引入教程型资料、博客型资料作为正式语料
- 私有协议 `SUT` 仅作为演示补充，不纳入 cleaned 标准语料

#### 语料目录结构已统一

当前数据结构已统一为：

- `data/protocols/raw/`
- `data/protocols/cleaned/`

这使得数据组织更清晰，也更利于论文中的复现性描述。

#### 新增 RFC 已补齐

目前已纳入并整理的新增 RFC 包括但不限于：

- HTTP 911x 系列
- TLS 扩展 / ALPN / PKI 相关 RFC
- IPv6 / PMTU 相关 RFC
- DNSSEC / RFC 2308 / RFC 1034 等 DNS 扩展文档
- 新版 TCP RFC 9293

相较于初始版本，知识库的协议覆盖面已经明显提升。

---

### 1.3 文档清洗规则已经过多轮迭代

这一部分已成为项目中具有较强论文价值的一部分。

当前已完成的清洗能力包括：

- 去除老 RFC 中的页眉页脚噪声
- 去除目录 / TOC
- 处理旧式下划线标题
- 处理现代 RFC 编号式目录页
- 处理 appendices / acknowledgments / author's address 等尾部内容
- 将大多数 cleaned 文档整理到适合检索与切块的状态

这部分可以在论文中明确表述为：

> 项目针对 RFC 文档年代跨度大、版式不统一、章节结构差异明显的问题，设计并迭代了轻量级清洗规则，以提升后续结构化切分与检索效果。

## 2. 距离最终论文还差什么

### 2.1 系统实现层面：还差少量结构整理

当前系统主功能已基本齐备，剩余的结构优化主要体现在：

- `app.py` 仍偏重
- QA 主流程编排尚未完全拆入 `src/qa_service.py`

这部分不是论文完成的决定性障碍，但如果时间允许，继续整理能增强工程性与可维护性。

---

### 2.2 实验部分：仍需“定稿化”

目前实验链路已经搭建完成，但距离最终论文可交付，还差：

- 人工评分结果的最终确认
- 图表的最终定稿
- 自动评测与人工评分结果的整合叙述
- 可能需要再补一组最小对比实验

这是当前离论文最近、也是最关键的工作部分。

---

### 2.3 对比实验仍略显不足

如果论文只展示“当前系统的效果”，说服力会略弱。

建议增加至少一组最小对比实验，例如：

#### 方案 A：查询改写前后对比

- 不做查询改写
- 当前两条子查询改写方案

#### 方案 B：chunk 策略对比

- `fixed`
- `hybrid`

其中，查询改写前后对比更贴合当前系统演进路径，实施成本也较低。

---

### 2.4 论文正文尚未完全成型

目前已经积累了大量系统、实验和图表材料，但这些内容仍是“模块化成果”，尚未完全整合为论文章节。

仍缺的主要是：

- 研究目标与问题界定的统一表述
- 方法章节与系统设计章节的完整组织
- 实验章节的正式成稿
- 总结与展望的整体收束

---

### 2.5 论文叙事链还需要进一步明确

最终论文需要回答的核心问题大致包括：

1. 为什么选择 RFC-first 的协议知识问答路线
2. 系统如何设计与实现
3. 针对 RFC 文档特点做了哪些关键适配
4. 系统的效果如何
5. 系统的优点与局限分别是什么

目前这些素材都已经具备，但还需要进一步整理成一条清晰的论文叙事链。

## 4. 建议的下一步工作计划

### 4.1 第一优先级：把实验部分做扎实

当前最值得优先推进的是实验部分，而不是继续做边缘工程优化。

建议优先完成：

1. 确认人工评分初稿是否接受
2. 定稿自动评测与人工评分图表
3. 补一组最小对比实验

---

### 4.2 第二优先级：形成实验结果分析定稿

在现有基础上，建议进一步整合形成正式实验章节，包括：

- 测试集说明
- 自动化评测结果
- 人工评分结果
- 图表分析
- 实验小结

这是最接近论文交付的部分，也最能直接提高完成度。

---

### 4.3 第三优先级：决定是否继续拆分 `qa_service`

这部分属于工程结构优化，重要但优先级已低于实验定稿。

- 如果时间较紧，可以先不做
- 如果仍有余力，可以作为系统实现章节的工程性补充

---

### 4.4 第四优先级：整合论文章节结构

建议逐步形成以下章节：

1. 绪论 / 研究背景
2. 相关技术与相关工作
3. 系统设计与实现
4. 数据处理与知识库构建
5. 实验设计与结果分析
6. 总结与展望

## 6. 阶段性结论

当前项目已经从“系统能否做出来”的阶段，进入到“如何把已有系统成果组织成一篇完整论文”的阶段。

因此，从现在开始，工作重心应更多放在：

- 实验定稿
- 图表定稿
- 结果分析定稿
- 正文结构整合

而不是继续花大量精力做边缘性工程优化。

换言之，当前最关键的任务已经不是“继续做系统”，而是“把系统成果转化为一份有说服力的论文成果”。
