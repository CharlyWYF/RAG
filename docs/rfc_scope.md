# RFC 文档范围说明

## 1. 项目定位

本项目面向“网络协议标准问答”场景，知识来源限定为 RFC（Request for Comments）标准文档，不引入教程型资料、博客或教材内容作为知识库语料。

系统目标是基于 RFC 原文实现面向协议标准的 RAG 问答，使回答具备以下特点：

- 基于标准文档内容生成
- 支持引用 RFC 编号与章节信息
- 在证据不足时进行保守回答或拒答
- 尽量降低幻觉并提升可追溯性

## 2. 文档选择原则

为控制本科毕业设计的实现复杂度，本项目不追求覆盖全部网络协议标准，而采用“小规模、高质量、可验证”的文档选择策略。

文档选择遵循以下原则：

- 优先选择经典、公开、结构清晰的 RFC 文档
- 优先选择适合构造问答任务的核心协议
- 优先覆盖不同网络层次，以保证问题类型多样性
- 控制首批文档数量，先完成核心系统与实验闭环，再考虑扩展

## 4. 分阶段纳入范围

### 4.1 第一批核心协议（主实验对象）

以下文档作为第一阶段系统构建、测试集设计、实验评估与论文主体分析的核心知识源：

- `tcp/rfc793.txt`
- `dns/rfc1035.txt`
- `http/rfc2616.txt`

选择理由如下：

- `TCP` 适合定义类、机制类、状态/流程类问题
- `DNS` 适合字段类、查询流程类、资源记录类问题
- `HTTP` 适合报文结构、状态码、方法语义类问题
- 三者覆盖面广，适合演示与实验设计
- 文档数量适中，便于优先完成清洗、切分、建库和问答闭环

### 4.2 第二批补充协议（增强覆盖）

在第一批协议运行稳定后，纳入以下文档以扩展系统覆盖范围：

- `ip/rfc791.txt`
- `udp/rfc768.txt`

选择理由如下：

- `IP` 可补充网络层报文结构、寻址和分片相关问题
- `UDP` 可补充与 `TCP` 的比较类问题
- 二者可增强协议对比类与字段类问答能力

### 4.3 第三批扩展协议（扩展实验对象）

在时间和实现条件允许的情况下，纳入以下文档作为扩展实验对象：

- `tls/rfc5246.txt`
- `tls/rfc8446.txt`
- `http/rfc7540.txt`

选择理由如下：

- `TLS 1.2` 与 `TLS 1.3` 可用于版本差异比较
- `HTTP/2` 可用于与传统 `HTTP/1.1` 做机制对比
- 适合作为扩展实验，不宜在第一阶段增加系统复杂度

### 4.4 暂缓纳入主实验范围的协议

以下文档暂不作为主实验对象，仅保留为后续补充或展示用途：

- `arp/rfc826.txt`
- `icmp/rfc792.txt`

原因如下：

- 文档篇幅相对较短
- 可构造的问题类型相对有限
- 对主体实验贡献不如 TCP、DNS、HTTP、IP、UDP 明显

## 5. 知识库范围约束

本项目知识库范围约束如下：

- 仅纳入 RFC 原文，不引入非标准解释性资料
- 回答应以检索到的 RFC 内容为依据
- 系统输出必须包含 RFC 编号和章节信息
- 当检索证据不足时，系统应避免给出确定性结论

## 6. 后续执行策略

后续文档处理与系统实现按以下顺序推进：

1. 优先处理 `TCP`、`DNS`、`HTTP`
2. 完成 RFC 清洗、结构切分、元数据设计与向量建库
3. 基于第一批协议构建测试问题集和实验基线
4. 再逐步加入 `IP`、`UDP`
5. 最后视时间情况扩展到 `TLS` 和 `HTTP/2`

## 7. 预期收益

采用上述范围控制策略后，可获得以下收益：

- 明确研究边界，避免知识源混杂
- 降低文档清洗与建库负担
- 提高回答引用的规范性与可信度
- 便于构建可复现、可评估、可答辩的毕业设计系统
