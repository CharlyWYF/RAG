# 本科毕业设计 Todo

## 项目主题
LLM-Based Knowledge Question-Answering System for Network Protocol Standards

## 总目标
- 完成一个基于 RAG 的网络协议标准问答系统
- 输出可演示的系统原型、实验结果、评估报告和最终论文
- 核心指标围绕 faithfulness、answer relevance、context relevance、latency、traceability

## 阶段一：开题后立刻完成的基础工作
- 明确研究边界：确定系统只聚焦“网络协议标准问答”，不扩展到通用网络知识
- 确定文档范围：选定首批协议文档，如 TCP、IP、HTTP、DNS、TLS 对应的 RFC
- 明确问题类型：定义系统回答哪些问题，如概念解释、字段含义、流程机制、参数约束、差异对比
- 明确评价维度：固定使用 faithfulness、answer relevance、context relevance、latency
- 和导师确认范围：确认是否只做 RFC，是否需要加入 IEEE 文档
- 建立项目仓库：整理 docs、data、src、experiments、report 目录
- 建立文献管理表：记录文献标题、年份、贡献点、与你课题的关系
- 建立项目日志：每周记录完成内容、问题、下周计划

## 阶段二：文献综述与需求分析
- 精读并整理 LLM 基础文献
- 精读并整理 RAG 核心文献
- 精读并整理 embedding、reranking、chunking 相关文献
- 整理“协议标准问答”领域现有研究缺口
- 总结为什么不能直接用普通 LLM
- 总结为什么 RAG 比微调更适合本科项目
- 输出一版文献综述框架：研究背景、技术演进、现有不足、你的切入点
- 形成需求文档：功能需求、非功能需求、输入输出定义、性能目标
- 画系统架构图：文档处理、向量检索、LLM 生成、前端展示四模块
- 写清楚成功标准：例如正确引用、回答相关、响应时间控制在 8s 内

## 阶段三：数据收集与知识库准备
- 确定协议文档清单，优先选公开、规范、结构清晰的标准
- 下载原始文档并统一保存命名规则
- 记录每份文档的来源、版本、发布时间
- 将 PDF、TXT、HTML 等格式统一转换为可处理文本
- 清理脏数据：页眉页脚、页码、目录噪音、乱码
- 提取元数据：协议名、RFC 编号、章节标题、小节编号
- 设计文档切分策略：按章节、按语义段落、按固定长度对比测试
- 实现 chunking 时保留上下文信息，如章节标题和协议标签
- 为每个 chunk 建立唯一 ID
- 构建知识库样例集，先用少量文档跑通流程
- 人工抽查 chunk 质量，确认没有切碎关键语义

## 阶段四：向量化与检索模块
- 调研可用 embedding 模型
- 选择一个主 embedding 模型，理由写清楚：性能、速度、资源消耗
- 建立向量生成脚本
- 建立向量库存储方案，优先考虑 FAISS
- 为每个 chunk 存储文本、向量、协议名、章节号、来源链接
- 实现基础 Top-K 检索
- 实现 metadata filter，例如只查某个协议
- 准备一批测试问题，观察召回结果
- 评估检索质量：能否召回正确章节、是否存在大量无关片段
- 对比不同 chunk 大小的检索效果
- 对比不同 Top-K 参数的效果
- 记录检索实验表格

## 阶段五：RAG 问答链路实现
- 设计用户输入格式
- 将问题转成 query embedding
- 从向量库召回相关 chunk
- 设计 prompt 模板，要求模型“必须基于检索内容回答”
- 在 prompt 中强制要求引用来源
- 增加“证据不足时拒答或保守回答”的规则
- 调整上下文拼接方式，避免 prompt 过长
- 测试不同 prompt 模板效果
- 实现 answer + citation 输出格式
- 为回答附上 RFC 编号、章节号、原文片段摘要
- 处理异常情况：查不到内容、检索结果冲突、输出无引用
- 对比 LLM-only 与 RAG 回答质量差异

## 阶段六：检索优化与幻觉控制
- 测试是否需要 reranking
- 如果时间允许，加入一个轻量 reranker
- 尝试 multi-query：把一个复杂问题拆成多个检索子问题
- 优化 chunk 排序策略
- 控制上下文数量，减少无关信息干扰
- 加入回答约束：没有证据不下结论
- 设计 hallucination 检查规则
- 记录典型错误案例：答非所问、引用不准、过度推断
- 针对错误案例迭代 chunking、prompt、Top-K 参数
- 固化最终 pipeline 配置，避免实验阶段频繁变动

## 阶段七：系统原型与前端展示
- 决定前端形式，建议 Streamlit
- 完成输入框、提交按钮、结果展示区
- 展示答案正文
- 展示引用来源
- 展示召回到的协议片段
- 展示响应时间
- 增加“未找到足够证据”的友好提示
- 处理多轮提问的基本交互
- 优化界面可读性，确保答辩演示清晰
- 准备演示用固定案例问题集

## 阶段八：实验设计与评估
- 建立测试问题集，覆盖不同问题类型
- 问题类型至少包括：定义类、流程类、字段类、约束类、比较类
- 为每个问题准备标准参考答案或参考证据
- 设计 baseline：纯 LLM 直接问答
- 设计实验组：RAG 问答系统
- 记录每个问题的回答、引用、响应时间
- 评估 faithfulness
- 评估 answer relevance
- 评估 context relevance
- 统计平均响应时间
- 对比不同参数设置的效果
- 做误差分析：哪些题型表现差、原因是什么
- 形成实验结果图表：柱状图、表格、案例对比图
- 总结系统优势与局限

## 阶段九：论文写作
- 先搭论文总目录
- 写第 1 章：研究背景、意义、目标与贡献
- 写第 2 章：相关研究与技术综述
- 写第 3 章：系统需求分析与总体设计
- 写第 4 章：方法设计与实现细节
- 写第 5 章：实验设计与结果分析
- 写第 6 章：总结与未来工作
- 整理图表编号、参考文献格式、术语统一
- 检查中英文摘要
- 检查图题、表题、引用格式是否符合学校要求
- 做重复率风险检查，避免综述部分表述过于贴文献原话
- 完成终稿排版与校对

## 阶段十：答辩准备
- 准备 PPT 初稿
- 提炼 3 个核心卖点：面向协议标准、基于 RAG、可追溯引用
- 准备系统演示视频或现场 demo
- 准备典型问答案例
- 准备“为什么不用微调”“为什么选 RAG”“如何降低幻觉”这类高频答辩问题
- 准备实验结果亮点页
- 准备局限性与未来工作页
- 进行 2-3 次模拟答辩

## 当前最关键的优先级
- 先确定协议文档范围
- 先做数据清洗 + chunking
- 先跑通 embedding + FAISS + Top-K retrieval
- 再接 LLM + prompt + citation
- 最后做评估 + 前端 + 论文

- 第 4 周：完成 embedding 与向量库
- 第 5 周：完成检索模块并做初测
- 第 6 周：完成 RAG 生成链路
- 第 7 周：优化引用与幻觉控制
- 第 8 周：完成前端原型
- 第 9 周：建立测试集并做实验
- 第 10 周：分析结果并补实验
- 第 11-12 周：集中写论文
- 第 13 周：改稿、查重、准备答辩
